# 文獻 J 資訊倫理與政策（Ethics & Policy）摘要

> 說明：本文件為「摘要版」（去除匯出對話資訊），聚焦「借閱隱私、兒少資料保護、紀錄保存年限、最小蒐集原則」。本文件非法律意見，實作前請與學校主管與法遵/資安窗口確認。

## 1) 核心概念與名詞（中英文對照表）
| 術語 (Term) | 一句話定義 | 同義詞/常見誤用 |
| --- | --- | --- |
| 資訊倫理（Information Ethics） | 關於資訊蒐集、使用、保存、分享的價值與規範 | 把倫理等同「不做任何記錄」而導致不可維運 |
| 個人資料（Personal Data） | 可識別自然人的任何資料 | 誤以為只有姓名才算個資；學號、行為紀錄也可能 |
| 敏感資料（Sensitive Data） | 若外洩會造成重大風險的資料（如借閱紀錄、兒少資訊） | 把借閱紀錄當一般資料公開查詢 |
| 目的限制（Purpose Limitation） | 只為明確目的蒐集與使用資料 | 先收再說，未定義用途與保存期限 |
| 最小蒐集（Data Minimization） | 只蒐集提供服務必要的資料 | 把校務系統欄位全搬進來 |
| 保存期限（Retention Period） | 資料可保存多久、何時刪除或彙總 | 沒期限＝永久保存，風險累積 |
| 存取控制（Access Control） | 控管誰能看/改哪些資料 | 只靠「大家自律」沒有系統限制 |
| 稽核紀錄（Audit Log） | 記錄敏感操作以利追溯 | 稽核紀錄存了太多個資或沒有保護 |
| 去識別/匿名化（De-identification/Anonymization） | 移除識別資訊或改寫成不可回推 | 只遮罩姓名但保留可回推的 ID 組合 |
| 資料外洩（Data Breach） | 未授權存取、外傳或被竄改 | 沒有事件通報流程與責任分工 |
| 兒少資料保護（Children’s Data Protection） | 對未成年資料採更高保護等級 | 把兒童借閱/搜尋紀錄當成活動評比資料 |
| 透明告知（Transparency Notice） | 告知蒐集目的、範圍、保存、權利與聯絡方式 | 只有一句「我們會保護你的資料」但沒有具體內容 |

## 2) 關鍵理論與必讀脈絡（為何重要）
- **借閱紀錄屬高敏感**：可推知興趣、信念、健康等；應預設最小保存與最小可見。
- **兒少情境放大風險**：校園資料常集中且具身分性；任何外洩都可能造成長期影響。
- **治理比功能更先**：沒有政策與權限，就算功能完整也不敢用、也不可長期維運。

## 3) 標準/規範/權威資源（可搜尋線索）
- 台灣《個人資料保護法》（搜尋：`個人資料保護法 借閱紀錄 學校`）
- 台灣《資通安全管理法》（公部門/學校資安治理脈絡，搜尋：`資通安全管理法 學校`）
- ALA Library Privacy Guidelines（搜尋：`ALA privacy guidelines circulation records`）
- ISO/IEC 27001（資訊安全管理體系）搜尋：`ISO 27001 information security management`

## 4) 中小學場域的簡化策略（做/不做/為什麼）
- **先做**：最小蒐集、角色分權、稽核紀錄、匯出與刪除/封存流程。
- **先不做**：過度細緻的行為追蹤與個人化推薦（除非能清楚說明目的與保護）。
- **歷史借閱預設不保留**：只保留進行中借閱與必要的彙總統計（避免永久留存）。
- **資料出口要有**：能匯出也能刪除/封存，避免資料成為「只能一直累積」的負債。

## 6) 對應到系統的功能設計建議（至少 8 條）
- 需求：降低個資風險 → 功能：最小蒐集欄位模板（匯入時提示不建議欄位）→ 為何：從源頭減少風險面。
- 需求：借閱隱私可控 → 功能：預設不保留借閱歷史；若保留需開關 + 保存期限 → 為何：符合目的限制與最小保存。
- 需求：限制可見範圍 → 功能：借閱紀錄只對授權角色可見；學生只能看自己的 → 為何：防止旁觀與濫用。
- 需求：可追溯濫用 → 功能：敏感資料查詢/匯出寫入稽核紀錄 → 為何：降低內部濫用、便於調查。
- 需求：可被刪除/封存 → 功能：定期清理（刪除或彙總）已到期的歷史紀錄 → 為何：讓系統可長期維運。
- 需求：降低資料外洩衝擊 → 功能：備份加密 + 權限分離 + 最小存取 → 為何：即使外洩也降低可用性。
- 需求：透明告知 → 功能：系統內建「隱私告知」頁面與版本紀錄 → 為何：提升信任與合規可用性。
- 需求：事件處置可執行 → 功能：建立事件通報流程（誰負責、何時通知、如何封存證據）→ 為何：避免出事才臨時應對。

## 7) 常見坑與避雷清單（至少 8 條）
- 借閱歷史預設永久保存，且可被多數人查詢。
- 為了報表需求直接匯出個人明細，檔案在校內傳來傳去。
- 把學生搜尋/借閱當成評比，造成不必要的監控感與壓力。
- 稽核紀錄沒有保護或可被刪改，反而成為風險點。
- 沒有保存期限與清理機制，資料越堆越多，外洩影響面變大。
- 權限只做 UI 控制，後端未限制，越權存取容易發生。
- 在公用電腦未登出或會話不過期，導致他人可操作館員功能。
- 事件發生時沒有通報與封存流程，延誤處理、難以釐清責任。

## 8) 可交付給 AI 小助理的任務清單（至少 10 條）
1. 產出「最小蒐集」欄位清單與「不建議蒐集」欄位清單（含理由）。
2. 擬定借閱紀錄保存政策方案（不保留/保留 N 天/只保留彙總）並列利弊。
3. 產出權限矩陣（角色 × 敏感資料 × 可見範圍）。
4. 撰寫隱私告知頁草案（目的、範圍、保存、權利、聯絡窗口）。
5. 設計稽核事件清單（哪些操作一定要記錄，metadata 存哪些欄位）。
6. 設計資料匯出規則（誰可匯出、匯出內容遮罩、匯出檔保存期限）。
7. 產出資料清理（Retention）批次作業規格（排程、對象、回報）。
8. 做一份「資料外洩事件處置流程」SOP（含角色分工）。
9. 針對自助借書/公用設備提出安全控管清單（自動登出、遮罩、權限）。
10. 盤點台灣校園情境可能適用的法規/政策清單（可搜尋線索）。
